01
MCP architecture hard
Is this actually MCP-compliant? The spec defines JSON-RPC 2.0 over stdio/SSE. Your McpServer is just an in-process Kotlin class.
▼
Short answer
No, it is not wire-protocol compliant — and that is intentional. EdgeMind implements the MCP conceptual model, not the transport layer.
Detailed explanation
The MCP spec assumes a network boundary between client and server — JSON-RPC 2.0 over stdio or SSE exists to cross that boundary. On a single device with no network surface, that transport is pure overhead. EdgeMind keeps the meaningful abstractions — named tools, scoped capability tokens, a server that enforces guardrails before routing — and replaces the wire format with direct Kotlin coroutine dispatch inside the same process. The argument is that the protocol's value is the orchestration pattern, not the serialisation format. If spec compliance were required, you would wrap McpServer.handleToolCall() in a JSON-RPC 2.0 handler over a local Unix socket — that is a one-file change and does not alter any of the architectural claims.
Summit framing: "We implement MCP semantics, not MCP transport. On a device with no network boundary, the transport layer is irrelevant — what matters is the orchestration contract."

Security & IAM hard
Your IamService.verifyToken() says "in production: verify EC signature" but doesn't actually verify it. So any JWT string passes?
▼
Short answer
Yes — signature verification is skipped in this demo. Token issuance is real, verification is payload-only parsing.
Detailed explanation
Token issuance is genuine — Android Keystore generates an EC P-256 keypair and the JWT is signed with SHA256withECDSA inside the hardware TEE. The private key never leaves the secure enclave. However, IamService.verifyToken() currently only parses the payload — it does not verify the signature. This is a deliberate demo simplification because the issuer and verifier are the same process: a forged token cannot arrive since there is no network surface. In a multi-process or multi-device scenario, the signature check is mandatory. The verifyToken() method exists precisely to mark where that fix goes — it is one call to the Android Keystore Signature.verify() API.
Known issue to fix before the summit: add Signature.getInstance("SHA256withECDSA").verify(signatureBytes) inside verifyToken(). Takes ~20 minutes.

Inference & perf hard
Your benchmark shows 180ms prefill for Phi-3 Mini at 128 tokens. Snapdragon 8 Gen 3 HTP peak is ~10 TOPS. Does the math check out?
▼
Short answer
Yes — 180ms is plausible when accounting for memory bandwidth bottlenecks, not just compute throughput.
Detailed explanation
Phi-3 Mini int4 is ~2.1 GB model weight. At 128 tokens, prefill is a matrix multiply over ~3.8B parameters quantised to 4-bit — roughly 1.9 GOPS. At 10 TOPS peak and accounting for memory bandwidth bottlenecks (LPDDR5X at ~77 GB/s on the SM8650), 180ms is realistic. The critical constraint is not compute but memory bandwidth — loading 2.1 GB of weights per forward pass at 77 GB/s gives ~27ms of pure memory load time, and the HTP has limited on-chip SRAM. The QAI Hub export targets SM8650 with HTP-specific op lowering and operator fusion that naive ONNX export would miss — that is where the 180ms number comes from, not raw TOPS arithmetic. Decode: 90ms / 64 tokens = ~1.4ms per token → ~714 tok/s decode throughput
Key insight: for LLM inference on-device, the bottleneck is almost always memory bandwidth (GB/s), not arithmetic throughput (TOPS). Know this distinction — it will come up.

Security & IAM hard
The GuardModel regex patterns are visible in the APK. An attacker can just avoid those exact strings and bypass the guard entirely.
▼
Short answer
Correct for the heuristic layer. The real guard is the 60M-parameter neural classifier which is opaque and cannot be pattern-matched against.
Detailed explanation
The architecture has two guard layers: (1) heuristicClassify() — regex patterns for obvious cases, intentionally transparent, fast pre-filter when native is unavailable. (2) nativeClassify() — a 60M-parameter .pte neural classifier compiled via ExecuTorch that runs in 22ms on the HTP. The 22ms latency is the giveaway — a rule-based filter runs in microseconds, not milliseconds. A neural classifier is an opaque function that does not pattern-match on strings, so knowing the regex patterns does not help bypass it. The security model also assumes the device owner is the attacker's adversary — prompt injection from a user of the app, not someone with APK access. APK-level attackers have already compromised the device.
Known issue: nativeClassify() is currently a stub. Before the summit, the real classifier needs to be wired in — otherwise the only guard is the visible regex layer, which is genuinely bypassable.

Training & LoRA hard
You call this federated learning. But there is no aggregation server, no secure aggregation, no differential privacy, and no global model improvement. That is just local fine-tuning.
▼
Short answer
You are correct. The accurate framing is "client-side FL" — the part of federated learning that runs on the device, without the aggregation server.
Detailed explanation
True FL (Google FedAvg, PySyft, Flower) requires: aggregation server, secure aggregation, differential privacy (noise injection), multiple participating devices, and global model improvement. EdgeMind has none of these. What it does match: data stays on device, model updates not raw data, PII protection via anonymise(), asynchronous training via WorkManager. The honest framing is: "This is on-device personalisation — the privacy-preserving end of the federated learning spectrum. Your device gets better from your interactions; nobody else benefits or contributes." The LoRA adapter could theoretically be shared as a model delta — that would be the aggregation step — but it is never is. Reserve "federated learning" for the full stack and use "client-side FL" or "on-device personalisation" instead.
Recommended summit framing: "federated learning without the federation — client-side only, maximum privacy, zero coordination overhead."

Inference & perf medium
Why use FAISS on Android rather than SQLite FTS5 with vector extension, or an ONNX-based ANN search?
▼
Short answer
FAISS gives exact control over index type and fits the 5k-vector demo scale at ~10MB. SQLite vector extensions are still experimental.
Detailed explanation
FAISS IndexFlatL2 over 5k vectors at 384 dimensions fits in ~10MB memory and returns results in 4ms on XNNPACK. The flat index is exact — no approximation error. For a clinical KB growing to 100k+ chunks, you would switch to an IVF index for sub-linear search time. SQLite vector extensions (sqlite-vss, sqlite-vec) are still experimental, lack index variety, and do not have stable arm64-v8a builds. The tradeoff is that FAISS requires a native .so build for arm64-v8a, whereas SQLite is already on-device. For a larger production system, SQLite + vss or a purpose-built on-device vector store would reduce the native dependency surface — but for this demo, FAISS is the right call. 5k vectors × 384 dims × 4 bytes = ~7.7 MB index size

What happens if the .pte files are not in assets? Does the app crash on launch?
▼
Short answer
No crash — every engine falls back to Kotlin mock implementations. The full demo runs on any Android 10+ device or emulator.
Detailed explanation
ExecuTorchRunner catches UnsatisfiedLinkError on System.loadLibrary("executorch_jni") and sets nativeAvailable = false. Every inference engine (SlmInferenceEngine, LlmInferenceEngine, GuardModel) checks runner.isReady() before calling native code and falls through to Kotlin mock implementations that return plausible responses. This means: No .pte files + no native libs → full mock demo on any Android 10+ device HTP available → real inference at published benchmark numbers The benchmark numbers on the slides are only valid on Snapdragon 8 Gen 3 HTP hardware. For the Mumbai summit, mock mode means presenting from any Android phone or even an emulator if the target device is unavailable.
Summit tip: always have mock mode ready as a fallback. Demo gods are real.

Governance medium
The HMAC-signed audit log is on-device in Room. How does it prevent the device owner from tampering with their own audit log?
▼
Short answer
It does not prevent a determined device owner from tampering — HMAC proves integrity of individual entries, not chain custody across a tampered database.
Detailed explanation
The HMAC-SHA256 signature on each audit entry proves that a specific entry was not modified after creation — if you change a field, the hash will not match. But a device owner with root access could delete entries from Room, or replace the entire database. The threat model here is: (1) proving to a regulator what happened, given a cooperative device, and (2) detecting accidental or app-level tampering. For stronger guarantees in regulated domains, you would add: remote attestation (Android SafetyNet/Play Integrity), log synchronisation to a write-only cloud endpoint, or a hardware-backed append-only log. The current design is appropriate for GDPR/DPDP audit trails where the device owner is also the data subject — not for adversarial audit scenarios.

Training & LoRA medium
LoRA training on 500 samples with rank 8 for 3 epochs — what is the actual memory footprint and how do you stay within 800MB?
▼
Short answer
Layer-by-layer fragmentation — only one transformer layer's activations and gradients are in memory at a time, not the full model.
Detailed explanation
Phi-3 Mini has 32 transformer layers. Full backprop would require storing activations for all 32 layers simultaneously — at 3.8B params int4 that is impractical on mobile. LoRA reduces the trainable parameter count dramatically: rank 8 adapters for each attention projection = ~8 × 8 × hidden_dim parameters per layer, roughly 0.1% of total params. Layer-by-layer fragmentation means: forward pass layer N → store only layer N activations → backward pass layer N → discard → move to N-1. Combined with gradient checkpointing (recompute activations on the backward pass instead of storing them), this keeps peak memory inside ~800MB on 12GB LPDDR5X devices. The 800MB ceiling is a soft constraint — WorkManager idle+charging condition also ensures the OS has not paged out memory to other apps. LoRA params: rank=8, alpha=16 → ~4M trainable params vs 3.8B total = 0.1%

Security & IAM easy
Why does GuardModel run before IAM validation, not after? Seems wasteful to classify prompts that might have an invalid token anyway.
▼
Short answer
Safety is more important than efficiency — a blocked prompt should never reach the token check, even on an invalid token path.
Detailed explanation
The ordering is deliberate: GuardModel → IAM → ToolRegistry → Handler. If IAM ran first: a valid token + blocked content would reach the guard. If the guard ran second: a window exists where a jailbreak prompt with a valid token could theoretically escape if the guard has a bug. Running the guard first means zero tokens are ever processed by Phi-3 or the 7B model if GuardModel blocks — even on the fast-path with a valid, elevated-scope token. The 22ms guard latency is less than a network round-trip to a cloud API, so the cost is negligible. This ordering also means blocked prompts never incur the IAM keystore lookup cost — a mild efficiency benefit for high-volume block scenarios.
The 22ms number is the summit punchline: "Our safety layer adds less latency than a network round-trip to a cloud guardrails API — and it runs before any LLM token is processed."

MCP architecture easy
What is the confidence threshold of 0.75 based on? Is it empirically validated or just a chosen constant?
▼
Short answer
It is a chosen constant in the current implementation — not empirically validated on a held-out clinical or domain-specific dataset.
Detailed explanation
The 0.75 threshold is a hyperparameter that controls the cost/latency vs quality tradeoff in the three-path routing. Below 0.75 → RAG fallback (adds ~50ms). Still below 0.75 after RAG → LLM escalation (adds 1-5s). A higher threshold escalates more aggressively to RAG/LLM, increasing quality but latency. A lower threshold keeps more in SLM, faster but potentially lower accuracy. The right value is domain-dependent: for medical triage you would want a higher threshold (less tolerance for uncertainty); for general chat you accept more SLM responses. The honest answer to this question is: "0.75 is a reasonable starting point — the on-device LoRA training loop will eventually learn domain-specific confidence calibration from accepted interactions, which is a better long-term answer than a fixed constant."

Hardware easy
Why does the Vulkan backend exist if HTP handles SLM and LLM inference? When does it actually fire?
▼
Short answer
Vulkan is a GPU fallback for the LLM path when HTP headroom is saturated — camera, gaming, or other NPU-intensive apps can exhaust HTP capacity.
Detailed explanation
The Snapdragon 8 Gen 3 HTP has limited concurrent capacity — if another app (camera computational photography, on-device speech recognition) has locked HTP resources, the LLM path falls back to Vulkan GPU compute. Vulkan is listed in AssistantViewModel.kt:229 as a secondary backend. It is slower than HTP for transformer workloads (no dedicated tensor instruction set) but faster than XNNPACK CPU for the 7B model. The compute backend priority is: HTP (primary) → Vulkan (GPU fallback) → XNNPACK (CPU last resort). XNNPACK handles RAG embeddings exclusively because MiniLM at 384 dimensions does not benefit from HTP's large-batch optimisation.

Governance easy
The Aadhaar pattern in GuardModel has false positive risk. What specific pattern are you using and why is it risky?
▼
Short answer
A 12-digit number regex will false-positive on phone numbers, order IDs, and timestamps — all common in clinical contexts.
Detailed explanation
Aadhaar numbers are 12 digits, typically formatted as XXXX XXXX XXXX. A naive regex like \d{12} or \d{4}\s\d{4}\s\d{4} will also match: 12-digit phone numbers (common in India), medical record numbers, timestamps in milliseconds, and long order or batch IDs. In a clinical context where patients dictate symptoms and a doctor types notes, the false positive rate could be high enough to block legitimate queries. The fix is a more specific pattern: Luhn-check validation (Aadhaar uses a Verhoeff checksum), prefix exclusion (valid Aadhaar starts with 2–9), and context windowing (surrounding words should not contain medical terms). Before the summit, either tighten the pattern or add a manual override for verified clinical users.
Known issue: Aadhaar regex false positive risk. Add Verhoeff checksum validation to reduce false positives in clinical text contexts.

Inference & perf hard
Greedy decoding produces repetitive outputs for longer LLM generations. Why not use beam search or sampling with temperature?
▼
Short answer
Greedy decoding is a deliberate choice for clinical safety — deterministic, reproducible, and auditable. Temperature sampling introduces non-determinism that conflicts with the audit trail story.
Detailed explanation
In a regulated medical context, the audit trail logs every inference decision with path, confidence, and latency. Temperature sampling means the same prompt produces different outputs on re-runs — breaking the reproducibility guarantee that makes the audit log meaningful. Greedy decoding (argmax at each step) is deterministic: given the same model weights and prompt, you always get the same output. This means: (1) you can reproduce any logged inference from the audit trail, (2) there is no variance in clinical advice that could be attributed to sampling randomness. Beam search would improve output quality at the cost of 2-5× latency per token — acceptable for an LLM path that is already 1-5s, but requires careful memory management. The current greedy approach is the right default for a regulated domain demo; beam search and top-p sampling are worthwhile future improvements.
